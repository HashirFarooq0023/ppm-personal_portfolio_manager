# Personal Portfolio Manager (PPSM) - Backend Architecture & Implementation Detail

This document provides an exhaustive, line-by-line detailed breakdown of the backend architecture for the PPSM project. It covers everything from the core FastAPI structure to the intricate data scraping logic and database management.

---

## 1. Core Architecture Overview

The PPSM Backend is built using **FastAPI**, a modern, high-performance web framework for building APIs with Python. It follows a decoupled service-oriented architecture where business logic is separated from the API routing layer.

### Technology Stack
- **Framework**: FastAPI (Asynchronous execution)
- **Database**: MongoDB (via Motor for async IO)
- **Authentication**: Clerk (JWT Verification with RS256)
- **Data Ingestion**: BeautifulSoup4 + HTTPX (Custom Scraper)
- **AI Engine**: OpenAI GPT-4o-mini
- **Deployment**: Render (Auto-scaling Web Service)

## 3. main.py: The Orchestrator

The `main.py` file is the heart of the backend. It initializes the FastAPI application and manages the background tasks that keep the market data fresh.

### Lifespan Manager
The `lifespan` async context manager handles startup and shutdown logic:
1. **Startup**: 
   - Calls `ensure_indexes()` to verify MongoDB performance constraints.
   - Launches `run_psx_scraper()` as a background task.
   - Launches `run_global_snapshot_task()` for historical data density.
2. **Shutdown**:
   - Gracefully cancels background tasks.
   - Waits for task cleanup before closing the app.

### Security Layer (Clerk Integration)
We implement a robust `get_current_user` dependency:
- **JWT Decoding**: Uses `python-jose` to verify tokens signed by Clerk.
- **JWKS Caching**: Public keys are fetched from Clerk's JWKS endpoint and cached to avoid excessive network calls.
- **RS256 Validation**: Ensures every protected request is authentically signed.

### Endpoint Catalog

#### /api/ping
- **Method**: GET
- **Purpose**: Health check and wakeup signal (essential for Render's free tier spin-up).

#### /api/portfolio
- **Method**: GET (Protected)
- **Purpose**: Returns the user's entire portfolio, calculated in real-time against live market prices.

#### /api/market/indices
- **Method**: GET (Public)
- **Purpose**: Returns current values for KSE-100 and KSE-30.

## 4. scraper.py: The Data Engine

The scraper is responsible for fetching real-time data from the Pakistan Stock Exchange (PSX).

### Selector Strategy
We target `https://dps.psx.com.pk/market-watch`. The scraper uses `BeautifulSoup` to parse the `DataTable` containing symbols, prices, and volumes.
- **Numeric Sector Mapping**: PSX often uses numeric IDs for sectors. We maintain a `PSX_SECTOR_MAP` to translate these (e.g., "0823" -> "TECHNOLOGY & COMMUNICATION").
- **Reliability**: The scraper includes exponential backoff for retries and layout change detection.

### Snapshot Logic
- **Real-time**: Prices are updated in the `market_watch` collection.
- **Historical**: Every 30 minutes, a global snapshot of all curated symbols is recorded in `market_history`. This allows for high-density candlestick chart generation in the frontend.

## 5. service.py: Business Logic & Persistence

The service layer contains the "brains" of the application.

### Database Connection Pool
We use `AsyncIOMotorClient` with optimized pooling:
- `maxPoolSize=10`: Prevents overloading Atlas Free Tier.
- `waitQueueTimeoutMS=5000`: Ensures the app fails fast rather than hanging on database lag.

### Key Logic Modules

#### ensure_indexes()
Defines the indexing strategy:
- `(symbol, time)` compound index for `market_history`.
- `clerk_id` index for the `portfolio` collection.
- `sector` index for market overview aggregation.

#### get_user_portfolio(clerk_id)
This is the most complex logic in the system:
1. Fetches all holdings for the user.
2. Bulk-fetches current market prices for all held symbols.
3. Calculates **Weighted Average Buy Price** and **Live P&L**.
4. Normalizes transaction IDs for consistent UI interactions.

#### generate_stock_analysis()
Leverages AI to provide investment insights:
- **Context Injection**: Feeds the user's current holdings and recent news into GPT-4o-mini.
- **Live Search**: Uses DuckDuckGo Search (`ddgs`) to fetch real-time news for specific symbols.
- **Financial Verdict**: Instructs the AI to provide a clear BUY/HOLD/SELL verdict.

## 6. models.py: Data Validation

Uses **Pydantic** for strict type checking and serialization.

### Core Schemas
- `PortfolioItem`: The internal database representation.
- `PortfolioSummary`: The unified response sent to the dashboard.
- `MarketIndex`: Structure for index tracking.
- `Transaction`: Detail for individual buy/sell events.

## 7. Database Structure (MongoDB)

### Collection: market_watch
Stores the most recent price for every tracked stock.
```json
{
  "symbol": "SYS",
  "current_price": 450.20,
  "change_percent": 1.5,
  "volume": 1200000,
  "sector": "TECHNOLOGY & COMMUNICATION",
  "last_updated": "ISODate"
}
```

### Collection: market_history
Intraday snapshots for chart generation.
```json
{
  "symbol": "KSE100",
  "value": 65120.45,
  "time": "ISODate"
}
```

### Collection: portfolio
User holdings and transaction ledgers.
```json
{
  "clerk_id": "user_25...",
  "symbol": "MEBL",
  "shares": 100,
  "average_buy_price": 185.50,
  "is_deleted": false,
  "transactions": [...]
}
```

## 8. Deployment & Scaling

### Render Web Service
- **Auto-Deploy**: Connected to the `main` branch of the GitHub repository.
- **Environment**: Python 3.11 with `pip install -r requirements.txt`.
- **Concurrency**: Managed by `uvicorn` with an asynchronous worker.

## 10. Development Guide

### Local Setup
1. Clone the repository.
2. Create a virtual environment: `python -m venv .venv`.
3. Install dependencies: `pip install -r backend/requirements.txt`.
4. Configure `.env` with `MONGODB_URI` and `OPENAI_API_KEY`.
5. Run the server: `python backend/main.py`.

### Testing
We use `pytest` for unit testing the service layer and `httpx` for integration testing the API endpoints. Ensure you use a test database to avoid data corruption.
